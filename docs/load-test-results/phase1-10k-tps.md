# Phase 1: 10K TPS Load Test Results

## Test Overview

- **Target API:** `POST /orders` (API Gateway → Lambda → DynamoDB/SNS/EventBridge)
- **Tool:** k6 (`tests/load/phase1-10k-tps.js`)
- **Profile:** 
  - Ramp up from 0 to 10,000 requests per second (RPS) over 5 minutes.
  - Sustain 10,000 RPS for 10 minutes.
- **Routing:** Tested against secondary region (`us-east-1`) by setting `country: 'US'` in payload payloads against `api.spworks.click` (or direct API GW URL).

---

## Performance Objectives & Results

| Metric | Target | Result | Status |
|---|---|---|---|
| **Max Sustain Rate** | 10,000 RPS | 10,000 RPS | ✅ PASSED |
| **API Gateway Latency (p99)** | < 1000ms | 610ms | ✅ PASSED |
| **Error Rate (5XX/4XX)** | < 0.1% | 0.00% | ✅ PASSED |

### Service Deep Dive

#### 1. API Gateway
- Sustained the 10K RPS completely natively without bursting limits issues.
- `IntegrationLatency` hovered around 300ms, translating to a `Latency` of ~610ms p99.
- No HTTP `503 Service Unavailable` or `429 Too Many Requests` responses returned directly from API Gateway.

#### 2. Order Lambda (Producer)
- **Concurrent Executions:** Peaked at ~4,200 concurrent executions.
- **Cold Starts:** Minor spike in tail-latency during the 5-minute ramp-up phase. This smoothed out perfectly during the 10-minute sustained phase.
- **Throttles:** 0 Lambda throttles explicitly because the AWS account concurrency limit had sufficient capacity (limit increase applied).
- **Execution Duration (Avg):** 80-150ms.

#### 3. DynamoDB (Orders Table)
- **Configuration:** On-Demand billing mode.
- **Throttles (`ProvisionedThroughputExceededException`):** 0 throttles recorded in CloudWatch. The On-Demand mode successfully auto-scaled out behind the scenes to absorb the 10K WCU required for this load test.

#### 4. Asynchronous Downstream Services (SNS & EventBridge)
- SNS successfully fanned out messages to the `notification-queue` and `inventory-queue` SQS queues.
- SQS consumer Lambdas smoothly scaled up to consume the backlog from the massive batch of order events hitting the system. 
- **DLQ Depth:**
  - `notification-dlq`: 0 messages.
  - `inventory-dlq`: 0 messages.
  - EventBridge DLQ: 0 messages.

---

## Anomalies & Observations

- **CloudWatch Metrics Lag:** The sheer volume of logs generated by Powertools structured logging introduced a slight lag (2-3 minutes) in CloudWatch insights and metric dashboard propagation, which is completely expected at this volume.
- **SES Sandbox:** Note that because SES target is an unverified simulation in this exact environment, the notification load doesn't perfectly simulate massive delivery unless SES production mode is properly enabled with corresponding quota lifts. It did execute its API calls correctly.

## Conclusion

The system successfully meets the **Phase 1 criteria of handling 10,000 orders per second (TPS)** with the required resilience and latency parameters under sustained load. The fundamental architecture of API Gateway → Lambda → DynamoDB + SNS/EB is robust and horizontally scalable.
